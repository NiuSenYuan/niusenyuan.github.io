Git 工作流程最佳实践

📅 2026-04-20 ✍️ 博主 ⏱ 6 分钟阅读
Git DevOps 工具

Git 是每个开发者的必备工具，但你真的用对了吗？本文将分享一些经过实践验证的 Git 工作流程和最佳实践。

提交信息规范

好的提交信息能让项目历史一目了然。推荐使用 Conventional Commits 规范：

类型(范围): 简短描述

详细描述（可选）

常用的类型前缀：

  • feat: 新功能
  • fix: Bug 修复
  • docs: 文档更新
  • style: 代码格式（不影响功能）
  • refactor: 重构
  • test: 测试相关
  • chore: 构建/工具相关

分支策略

Git Flow

经典的 Git Flow 分支模型：

  • main: 生产环境代码
  • develop: 开发主线
  • feature/*: 功能分支
  • release/*: 发布分支
  • hotfix/*: 紧急修复分支

GitHub Flow（推荐用于小团队）

更简单的流程：

  1. 从 main 创建功能分支
  2. 在功能分支上开发和提交
  3. 发起 Pull Request
  4. 代码审查后合并到 main
  5. 部署 main 分支

实用技巧

交互式暂存

git add -p  # 逐个确认要暂存的更改

修改最后一次提交

git commit --amend  # 修改提交信息或补充文件

查看优雅的提交历史

git log --oneline --graph --all --decorate

暂存工作进度

git stash       # 暂存当前修改
git stash pop   # 恢复暂存的修改

解决合并冲突

当合并冲突时，Git 会在文件中标记冲突区域：

<<<<<<< HEAD
你的修改
=======
对方的修改
>>>>>>> feature-branch

手动编辑文件，删除冲突标记，保留想要的内容，然后 git add 标记为已解决。

建议：频繁拉取远程代码（git pull --rebase），减少大规模冲突的概率。使用 rebase 而不是 merge 可以保持更清晰的历史。

总结

良好的 Git 实践能让团队协作更高效。记住：清晰的提交信息、合理的分支策略、频繁的代码同步，是高效使用 Git 的三大基石。
